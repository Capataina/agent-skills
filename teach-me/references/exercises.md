# Exercise Generation Reference

This file defines the requirements, templates, and standards for all exercises generated by the teach-me skill. Exercises are the practical counterpart to concept files — they prove understanding through implementation, not just reading.

## Core Rule: Exercises Are Skeletons

This is the single most important rule in exercise generation. Every implementation exercise must be a skeleton with TODOs, never a complete implementation. Providing working code defeats the entire purpose of the exercise.

What a skeleton includes:
- Function signatures with type hints
- Comprehensive docstrings with step-by-step hints
- Expected output for specific test inputs
- Time estimates
- `pass` or `# TODO` placeholders where the implementation goes
- Test cases at the bottom (also as skeleton or with expected values documented)

What a skeleton does not include:
- Any working implementation code
- Filled-in function bodies
- Comments like "# This is where the bug is" in debugging challenges

## Exercise Distribution

Target 15–20 total exercises for a medium-sized project. Distribute them across four categories:

| Category | Count | Difficulty Mix | Purpose |
|---|---|---|---|
| Foundations | 3–4 | Mostly easy, 1 medium | Build confidence with basic concepts |
| Core domain | 5–7 | Mix of easy and medium | Master the main algorithms and patterns |
| Advanced concepts | 3–5 | Mostly medium, 1–2 hard | Integrate multiple concepts |
| Domain patterns | 2–4 | Medium to hard | Apply reusable patterns |

### Difficulty Levels

- **Easy** (30% of exercises): 15–30 minutes, single concept, clear solution path. The reader should be able to complete this after reading one concept file.
- **Medium** (50% of exercises): 30–60 minutes, multiple concepts, some ambiguity in approach. Requires understanding interactions between concepts.
- **Hard** (20% of exercises): 1–2 hours, integration across systems, debugging, or design decisions. Requires holistic understanding.

### Skeleton Checklist

Before finalising any implementation exercise, verify:
- [ ] Function signatures are present with type hints/annotations
- [ ] Docstring includes a description of what to implement
- [ ] Docstring includes numbered step-by-step hints
- [ ] Docstring includes expected output for at least 2 specific test inputs
- [ ] Docstring includes a time estimate
- [ ] Docstring includes difficulty level
- [ ] Docstring includes prerequisite concept file references
- [ ] Function body contains only `pass`, `todo!()`, or `# TODO` comments
- [ ] Test cases at the bottom show expected values (can be commented out)
- [ ] No working implementation code anywhere in the file

### 2. Debugging Challenges (`exercises/debugging_challenges/`)

Broken code that the reader must diagnose and fix. These exercises build debugging skills and deepen understanding of failure modes.

**Critical rules:**
- The code must actually be broken in a realistic, non-obvious way
- No comments or variable names that hint at the bug location
- The bug should be conceptual (not a typo or syntax error) — something that reveals misunderstanding of the underlying concept
- Include enough working code around the bug that the reader can run it and observe the failure

**Template:**

````python
"""
DEBUGGING CHALLENGE: Diverging Policy Gradient

This A2C implementation diverges after approximately 100 episodes. Find and fix the bug.

Expected behaviour (when fixed):
  - Running reward should increase steadily
  - Should solve CartPole (reward > 195) within 500–800 episodes
  - Loss values should remain bounded

Actual behaviour (with bug):
  - Reward is erratic: spikes and crashes
  - May produce NaN losses after a few hundred episodes
  - Policy becomes degenerate (always picks the same action)

Hints (read these in order if stuck):
  1. The bug is in the gradient computation, not the network architecture
  2. Think about what happens when returns have very high variance
  3. Policy gradient algorithms almost always require a specific preprocessing step on the returns

Time estimate: 30–45 minutes
Difficulty: Medium
Prerequisites: concepts/core/reinforcement_learning/policy_gradients.md
"""

import numpy as np


class PolicyNetwork:
    def __init__(self, input_dim, output_dim, lr=0.001):
        self.weights = np.random.randn(input_dim, output_dim) * 0.1
        self.lr = lr

    def forward(self, state):
        logits = state @ self.weights
        exp_logits = np.exp(logits - np.max(logits))
        return exp_logits / exp_logits.sum()

    def select_action(self, state):
        probs = self.forward(state)
        action = np.random.choice(len(probs), p=probs)
        return action, np.log(probs[action])


def compute_returns(rewards, gamma=0.99):
    returns = np.zeros_like(rewards, dtype=np.float64)
    G = 0.0
    for t in reversed(range(len(rewards))):
        G = rewards[t] + gamma * G
        returns[t] = G
    return returns


def train_episode(policy, states, actions, rewards, gamma=0.99):
    returns = compute_returns(rewards, gamma)

    # Update policy using REINFORCE
    for t in range(len(states)):
        state = states[t]
        action = actions[t]
        G_t = returns[t]

        probs = policy.forward(state)
        grad_log = np.zeros_like(probs)
        grad_log[action] = 1.0 / probs[action]

        # Policy gradient update
        policy.weights += policy.lr * G_t * np.outer(state, grad_log)


# To test: create a simple environment and run for 500 episodes
# The reward should increase steadily if the bug is fixed
````

**Bug design principles:**
- The bug above is that returns are not normalised before being used as gradient weights. Raw returns can range from 10 to 500+, creating unstable gradients.
- Good bugs test understanding of the concept, not just code literacy.
- Each debugging challenge should target a specific misconception from the concept file's misconceptions section.

### 3. SOLUTIONS.md

Include a `SOLUTIONS.md` file alongside debugging challenges with progressive hints. The reader should try each hint level before moving to the next.

````markdown
# Exercise Solutions and Hints

---

## Debugging Challenge 1: Diverging Policy Gradient

### Hint 1 (try this first)
The bug is in how returns are used in the gradient computation. Review the mathematical formula for policy gradients and compare it to the code.

### Hint 2 (if still stuck after 10 minutes)
Compare the variables used in the gradient computation to the standard REINFORCE formula. Returns in this implementation can range from ~10 to ~500. What effect does this have on gradient magnitude?

### Hint 3 (if still stuck after 20 minutes)
Returns should be normalised to zero mean and unit variance before being used as advantage weights. This ensures consistent gradient scales across episodes regardless of the absolute reward magnitude.

### Full Solution (only read after solving or giving up)
The bug is in `train_episode()`. Returns are used directly without normalisation:

```python
# Buggy: raw returns used directly
returns = compute_returns(rewards, gamma)

# Fixed: normalise returns before use
returns = compute_returns(rewards, gamma)
returns = (returns - returns.mean()) / (returns.std() + 1e-8)
```

**Why this matters:** Raw returns range from approximately 10 (early termination) to 500+ (successful episodes). This means gradient magnitudes vary by 50× between episodes. Early in training, when the agent occasionally succeeds, the large gradient from a high-return episode can destabilise the entire policy. Normalisation ensures every episode contributes proportionally, regardless of absolute reward scale.

**Conceptual connection:** This is the same reason batch normalisation works in supervised learning — keeping activations (or in this case, gradient weights) on a consistent scale prevents training instability.
````

### 4. Other Exercise Types (Optional)

Create these when the project warrants them:

**Comparison Challenges** (`exercises/comparison_challenges/`):
- "Implement both X and Y, run them on the same input, compare results"
- Good for building intuition about algorithmic trade-offs
- Template: two skeleton functions side-by-side, a comparison harness at the bottom

**Optimisation Challenges** (`exercises/optimization_challenges/`):
- "This working implementation is correct but slow. Make it 10× faster"
- Provide a working (but inefficient) implementation, performance benchmark, and target speedup
- Good for systems-oriented projects

**Extension Challenges** (`exercises/extension_challenges/`):
- "This working code does X. Extend it to also do Y"
- Provide a working base implementation and a specification for the extension
- Good for understanding how systems compose

## Exercise Language Selection

Match the exercise language to the project's primary language:
- Rust project → Rust exercises (with Python alternatives for algorithm-focused exercises where Rust boilerplate would obscure the concept)
- Python project → Python exercises
- C++ project → C++ exercises
- Mixed project → Use the language that best fits the concept being exercised

For foundational/algorithmic concepts (sorting, gradient computation, tree traversal), Python is often clearest regardless of project language, because it minimises syntactic overhead and lets the reader focus on the algorithm.

For systems concepts (lock-free data structures, memory management, async patterns), use the project's language so the reader practises with the actual idioms they will encounter in the codebase.

## Exercise Cross-Referencing

Every exercise should reference:
1. The concept file(s) that teach the prerequisite knowledge
2. The project-specific implementation file that shows how the concept is used in the codebase
3. Related exercises that build on or complement this one

Include these references in the docstring under a "Prerequisites" or "Related" heading. This turns the exercise set into a connected graph rather than an isolated list.
