## Concept File Template

Use this template for files in `concepts/foundations/`, `concepts/core/`, and `concepts/advanced/`.

**Section inclusion logic:**
- **Always include**: Prerequisites, Target Depth, Core Concept, Worked Examples (3–5), Common Misconceptions (5–8), Glossary (15–20 terms), Recommended Materials (8–12 resources)
- **Include if applicable**: Mathematical Foundation (for math-heavy concepts), Step-by-step progressive build-up (for complex concepts)
- **Can merge**: "What Problem Are We Solving?" can be folded into Core Concept if it would be redundant as a separate section

````markdown
# [Concept Name]

## Prerequisites
Before learning this, you should understand:
- [Prerequisite 1] (Level X understanding required)
- [Prerequisite 2] (Level X understanding required)

## Target Depth for This Project
🎯 **Level X** — [Brief description of what this level means for this concept]

Levels:
- Level 1: Can use as a black box
- Level 2: Can explain intuitively to a non-technical person
- Level 3: Can explain with technical depth, understands the mathematics/architecture
- Level 4: Can implement from scratch and debug issues
- Level 5: Research-level mastery, can propose modifications

## What Problem Are We Solving?
[Real-world motivation — why does this concept exist? 2–3 sentences maximum. Can be folded into Core Concept if doing so is cleaner.]

## Core Concept
[Intuitive explanation with concrete examples. This is the heart of the file. Start with the simplest possible framing, then build up.]

[For complex concepts, use progressive build-up:]

### Step 1: [Foundational Sub-Concept]
[Build up from basics with concrete examples. The reader must fully understand this before moving on.]

### Step 2: [Intermediate Sub-Concept]
[Build on Step 1. Reference specific ideas from Step 1 explicitly.]

### Step N: [Advanced Synthesis]
[Combine all preceding steps into the complete concept. Show how the pieces fit together.]

## Mathematical Foundation
[Only include if the concept requires mathematics. Show full derivations with every intermediate step. Include worked numerical examples for every formula.]

[Skip this section entirely if the concept has no mathematical component.]

## Worked Examples

[Generate 3–5 worked examples. Each must show the concept from a different angle. Work through every step — do not skip intermediate calculations or assume the reader can fill in gaps.]

### Example 1: [Basic Mechanics]
[Show the core computation with simple, concrete numerical values. Walk through every intermediate step.]

### Example 2: [Parameter Variation]
[Show what happens when you change a key parameter. Build intuition for sensitivity.]

### Example 3: [Failure Mode or Edge Case]
[Show what goes wrong and why. This reveals the boundaries of the concept.]

### Example 4: [Comparison to Alternative] (if applicable)
[Compare this approach to a closely related one. Show why one is preferred in this context.]

### Example 5: [Connection to Project Code] (if applicable)
[Show how this concept manifests in the actual project, referencing specific files or functions.]

## How [Project Name] Uses This
[Brief explanation of where and how this concept appears in the project code. 2–4 sentences.]

[Reference: See `project_specific/implementations/xyz.md` for detailed implementation.]

## Common Misconceptions

[Generate 5–8 misconceptions. Cover beginner mistakes, persistent intermediate errors, failure mode misunderstandings, and confusions with related concepts.]

❌ **Misconception 1**: [Wrong belief, stated clearly]
✅ **Reality**: [Correct understanding with enough explanation to show why the misconception is wrong]

❌ **Misconception 2**: [Wrong belief]
✅ **Reality**: [Correct understanding]

[Continue for 5–8 total misconceptions]

## Glossary

[Define 15–20 terms used in this file. Define everything that is not plain English — even terms that seem obvious to an expert.]

### [Term 1]
[1–3 sentence definition with a concrete example]

### [Term 2]
[Definition] (For comprehensive definition, see [GLOSSARY.md#term2])

[Continue for all technical terms used in this file]

## Recommended Materials

[Provide 8–12 resources organised by type. Follow the resource entry format from writing-standards.md.]

### Textbooks (2–3 resources)
- [ ] **[Book Title] — [Specific Chapter/Section]**
  - **Read**: [Specific pages or sections]
  - **Skip**: [Sections not relevant or covered elsewhere]
  - **Focus on**: [Key theorem, derivation, or concept]
  - **Exercise**: [What to do after reading]
  - **Why**: [1 sentence on unique value]
  - **Time**: [X hours] | **Difficulty**: [Beginner/Intermediate/Advanced]
  - **Link**: [URL if available]

### Videos (2–3 resources)
- [ ] **[Lecture/Video Title]**
  - **Watch**: [Full lecture or specific timestamp range]
  - **Key timestamps**: [Timestamp 1] ([topic]), [Timestamp 2] ([topic])
  - **Why**: [1 sentence on unique value]
  - **Time**: [X minutes] | **Difficulty**: [Beginner/Intermediate/Advanced]
  - **Link**: [URL]

### Blog Posts / Articles (2–3 resources)
- [ ] **[Article Title]**
  - **Read**: [Full post or specific sections]
  - **Focus on**: [Key sections or insights]
  - **Why**: [1 sentence on value]
  - **Time**: [X minutes] | **Difficulty**: [Beginner/Intermediate/Advanced]
  - **Link**: [URL]

### Papers (1–2 resources, if relevant)
- [ ] **[Paper Title] — [Authors, Year]**
  - **Read**: [Specific sections, e.g. "Sections 1–3"]
  - **Skip**: [Sections too advanced or not relevant]
  - **Why**: [E.g. "The original paper" or "Best mathematical treatment"]
  - **Time**: [X hours] | **Difficulty**: Advanced
  - **Link**: [arXiv or journal URL]

### Interactive / Code (1–2 resources)
- [ ] **[Tool/Repository Name]**
  - **What to do**: [E.g. "Run the notebook", "Compare with project code"]
  - **Why**: [1 sentence]
  - **Time**: [X minutes] | **Difficulty**: [Level]
  - **Link**: [URL]
````

## Domain Pattern File Template

Use this template for files in `concepts/domain_patterns/`. Domain patterns are reusable architectural patterns that transfer across projects — the most valuable knowledge in the curriculum for long-term career utility.

````markdown
# [Pattern Name]

## What Is This Pattern?
[1–2 sentence definition of the pattern. Focus on what problem it solves and in what context.]

## When To Use This Pattern

✅ **Good for:**
- [Use case 1]
- [Use case 2]
- [Use case 3]

❌ **Not good for:**
- [Anti-pattern 1 — explain why]
- [Anti-pattern 2 — explain why]

## Core Concept
[Explain how the pattern works with a simple example — can be pseudocode or conceptual. Start from the simplest possible formulation, then build up.]

## Key Design Decisions

[Most patterns involve trade-offs. Explain each major decision point.]

### Decision 1: [Trade-off name]
- **Option A**: [Approach 1]
  - Pros: [...]
  - Cons: [...]
- **Option B**: [Approach 2]
  - Pros: [...]
  - Cons: [...]
- **When to use each**: [Clear guidance]

### Decision 2: [Trade-off name]
[Same structure]

## Worked Examples

[Generate 3–5 examples showing the pattern in different contexts and scales.]

### Example 1: [Basic Implementation]
[Show the pattern in its simplest form with concrete code or pseudocode]

### Example 2: [Variation or Edge Case]
[Show how the pattern adapts to different requirements]

### Example 3: [Real-World Usage]
[Show how a known open-source project or well-known system uses this pattern]

## Simplified Reference Implementation

[Include a minimal (20–80 line), self-contained code example showing the pattern. This should be clear enough to serve as a starting point for reimplementation.]

```python
# Example showing the core pattern
# This is a reference, not an exercise — include the full working implementation
```

## How [Project Name] Implements This
[Brief overview of this project's specific implementation. 2–4 sentences.]

[For implementation details: See `project_specific/implementations/xyz.md`]

## Variations

[List the major variations of this pattern and when to use each.]

- **Variation 1**: [Name] — [When/why to use, 2–3 sentences]
- **Variation 2**: [Name] — [When/why to use, 2–3 sentences]
- **Variation 3**: [Name] — [When/why to use, 2–3 sentences]

## Common Pitfalls

[5–8 common mistakes when using this pattern]

❌ **Pitfall 1**: [Mistake description]
✅ **Solution**: [How to avoid or fix it]

❌ **Pitfall 2**: [Mistake description]
✅ **Solution**: [How to avoid or fix it]

[Continue for 5–8 pitfalls]

## Projects That Use This Pattern

[Show that this pattern is widely used and transferable. 3–5 examples from real systems.]

- **[Project/System A]**: [How they use it, 1–2 sentences]
- **[Project/System B]**: [How they use it, 1–2 sentences]
- **[This Project]**: [How we use it, 1–2 sentences]

## Glossary
[15–20 pattern-specific terms]

## Recommended Materials
[8–12 resources focused on the pattern itself, using the standard resource entry format]
````

## CURRICULUM.md Template

````markdown
# Learning Curriculum: [Project Name]

## Overview
[2–4 sentences describing what this curriculum covers and what the reader will be able to do after completing it.]

**Estimated total time**: X–Y weeks (assuming Z hours per week)
**Learning path**: [Linear / Flexible / Modular]

---

## How To Use This Curriculum

- ✅ Check off items as you complete them
- ⭐ Star indicates critical path (must understand for interviews and code modification)
- 🔵 Blue circle indicates optional (valuable but not blocking)

**Different learning styles:**
- **Bottom-up learner**: Start with Phase 1, work sequentially
- **Top-down learner**: Read `project_specific/architecture_decisions.md` first, then backfill concepts as needed
- **Hands-on learner**: Jump to exercises, read concept files when you get stuck

---

## Phase 1: [Phase Name] (Est. X weeks)

### Learning Goals
After this phase, you will be able to:
- [Goal 1 — concrete, verifiable]
- [Goal 2 — concrete, verifiable]
- [Goal 3 — concrete, verifiable]

### Prerequisites

- [ ] [Prerequisite 1]
- [ ] [Prerequisite 2]

### Concepts to Learn

### Materials

### Exercises

### Phase Milestone

Examples of good milestones:
- Implement [X] from scratch without looking at reference code

After completing this curriculum, you should be able to confidently answer:

1. **"How does [Project] work?"**
   [Suggested 3–5 minute explanation structure]

   [Key concepts that often come up in interviews]
````
